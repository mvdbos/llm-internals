Lesson 1: Tensors, Parameters, and Runtime State

A language model is not one giant table of facts. It is a computation that moves shaped arrays of numbers through a sequence of operations. To reason about memory, first separate the numbers stored before a run from the numbers created during a run.

1. A tensor has shape, data type, and role

A tensor is a multidimensional array. Rank describes how many axes it has; shape gives the length of each axis; data type describes how each value is represented. None of those tells you the tensor's role.

A parameter is a learned value in a checkpoint. An activation is produced by executing the model. Retained runtime state is the subset of runtime values deliberately kept for later generation steps. Calling all three “the model” hides the memory boundary that matters.

2. What is stored, and what runs?

A model artifact stores parameter tensors and metadata. An inference engine reads that artifact, allocates working memory, executes the architecture, and applies a decoding rule. Artifact size is therefore not the same as peak runtime memory.

Before the run
Checkpoint-derived parameter tensors and metadata
Loaded
Parameters are mapped, copied, aligned, or unpacked as required
During the run
Activations, retained state, and workspaces are allocated
Measured peak
All allocations inside the declared boundary

3. Follow one next-token step without skipping stages

Architecture scope: the following trace is a conventional decoder-only, Transformer-style model. It is a useful concrete example, not a claim that every language model uses full self-attention or a KV cache.

  1. Text becomes IDs. A tokenizer maps text into token IDs. The exact mapping belongs to the checkpoint/tokenizer revision.
  2. IDs become vectors. An embedding layer looks up one learned vector per token ID. For a deliberately tiny example, suppose the selected embedding is e = [1, −2] with shape (2,). A learned projection W has shape (3, 2): 
    W = [[ 0.50,  0.25],
     [-1.00,  0.50],
     [ 2.00, -0.50]]

W × e = [0.50×1 + 0.25×(−2),
         −1.00×1 + 0.50×(−2),
          2.00×1 − 0.50×(−2)]
      = [0, −2, 3]                 shape (3,)
    The matrix's second dimension must match the vector's only dimension; its first dimension determines the output length.
  3. Vectors become contextual representations. Repeated blocks transform the hidden state. A conventional block combines normalization—for example, RMSNorm in some architectures—causal attention, residual paths, and a position-wise feed-forward network. Exact order, normalization variant, dimensions, gating, and attention pattern are architecture-specific (canonical architecture sources).
  4. Representations become scores. An output projection produces logits, one raw score per vocabulary item for the next position.
  5. A decoding rule selects an ID. Greedy choice, sampling, constraints, or another rule chooses the next token. The tokenizer decodes IDs back to text.
  6. The next step reuses state. Conventional cached attention appends new key/value vectors and reuses old vectors. It does not store attention scores as the cache and does not normally rewrite every old entry.

Toy round trip: text "blue" is mapped by a fictional tokenizer to ID 17; ID 17 selects an embedding; projections and repeated blocks transform it with the preceding context; the final vector produces vocabulary logits; a decoding rule selects ID 42; and the same fictional tokenizer maps ID 42 to " sky". Real models use much larger vectors and many blocks, but every arrow names a distinct object rather than skipping from text directly to an answer.

4. Architecture labels change the memory formula

Conventional cached attentionKeys and values per retained KV head, layer, and positionHow many KV heads, layers, positions, and bytes per element?
Grouped- or multi-query attentionFewer retained KV heads than query headsDid the calculation use KV-head count rather than query-head count?
Windowed/hybrid attentionDifferent retained lengths or mechanisms by layerWhich layers retain a window, global tokens, or full history?
Recurrent, state-space, or linear-attention pathArchitecture-specific recurrent or summary stateWhat state tensors does this architecture actually retain?

The safe habit is simple: name the architecture family, read the configuration, then write the formula. Do not start with “two bytes per parameter plus some cache.”

5. Static memory and context-dependent memory

Static memory is the loaded parameter allocation under a stated loading policy. Runtime memory also includes retained state, temporary activations/workspaces, and runtime overhead.

peak runtime memory
≈ loaded parameter allocation
+ architecture-specific retained state
+ temporary activations and workspaces
+ runtime/allocator overhead

Loaded parameter memory is mostly independent of prompt length. Retained state and many temporary allocations are not. Increasing batch size, retained sequence length, or generation strategy can change the peak even when the artifact is unchanged.

Worked separation

Suppose two runs load the same parameter tensors. Run B processes a longer retained sequence than Run A. If B uses more memory, that does not mean its parameter tensors grew; the context-dependent part changed. To diagnose it:

  1. measure loaded idle memory;
  2. record batch, prompt length, generated length, and cache/state data type;
  3. measure peak generation memory;
  4. attribute the difference only after checking retained state, workspaces, and allocator behavior.

6. What weight-only quantization changes

In a weight-only conversion, the quantized target is selected learned parameter tensors in the artifact. Their stored numeric representation and reconstruction metadata change. The tokenizer, architecture graph, prompt, decoding rule, temporary activation format, and retained-state format do not change merely because the weights changed; any of those would require a separate configuration decision. Learned behavior is primarily encoded in parameters—including the embedding lookup—but observed behavior also depends on architecture, tokenizer, prompt, decoding, and external context or tools.
