Lesson 2: How Quantization Actually Works

From 16-bit floats to 4-bit integers — and why the model still works. ~7 minutes.

What quantization does

Every weight in an LLM is stored as a number. At FP16, each number uses 16 bits (2 bytes). Quantization says: "what if we stored each weight in fewer bits?"

FormatBits per weightDistinct valuesAnalogy
FP161665,536 (in any range)Measuring to the millimeter
Q8 (8-bit)8256Measuring to the centimeter
Q4 (4-bit)416Measuring to the nearest 10 cm
Q2 (2-bit)24Measuring to the nearest half-meter
ELI5 version: You have a photo. The original has millions of colors (FP16). You convert it to 256 colors (Q8) — it looks almost the same. You convert to 16 colors (Q4) — it looks slightly posterized but still recognizable. You convert to 4 colors (Q2) — now it's barely recognizable. Quantization is the same tradeoff for model weights.

The core trick: scale factors

You can't just round 0.0234 to 0 and -0.891 to -1 and call it done. The trick is: find the min and max value in a group of weights, then map that range to your available integer buckets.


Step 1: Look at a group of weights:  [0.02, -0.45, 0.31, -0.88, 0.15, -0.02, 0.67, -0.33]
Step 2: Find min = -0.88, max = 0.67
Step 3: Scale: 0→15 maps to -0.88→0.67

  Int 0  →  -0.88    (min)
  Int 4  →  -0.49
  Int 8  →  -0.10
  Int 12 →   0.29
  Int 15 →   0.67    (max)

Step 4: Store the 8 weights as 4-bit ints + one FP16 scale factor

The key insight: the scale factor is stored in full precision. So each group of weights shares one high-precision scale and offset, and the individual weights are just cheap integers relative to that scale. This is called block-wise or group-wise quantization.

🧠 Why this works

Weights within a small group tend to be in a similar range. A single scale factor can represent them all well. The model trained with those weights already "expects" small variations — a little fuzziness per weight doesn't propagate catastrophically. It's like audio compression: you lose some fidelity, but the song is still recognizable because you kept the overall structure (the scale per block).

Decoding the GGUF suffixes

When you download models from Hugging Face, you see filenames like Q4_K_M, Q8_0, IQ3_XXS. Here's what they mean:

SuffixBits/weightSize (7B model)What it does
F1616~14 GBFull half-precision. No quantization. The quality baseline.
Q8_08~7 GB8-bit with block size 32. Minimal quality loss (1-2%).
Q6_K6~5.3 GB6-bit with K-quant superblock. Very good quality.
Q5_K_M5~4.5 GB5-bit K-quant, medium size. Good balance point.
Q4_K_M4~3.8 GB4-bit K-quant, medium. The community sweet spot.
IQ4_XS4~3.5 GB4-bit importance-weighted. Smarter than Q4_K_M for some models.
Q3_K_M3~3.1 GB3-bit. Quality loss becomes noticeable for reasoning.
IQ3_XXS~3~2.9 GBAggressively compressed. Noticeable quality loss on complex tasks.
Q2_K2~2.5 GBVery aggressive. Usable only for simple tasks or tiny models.

What do the letters mean?

LetterMeaning
QTraditional linear quantization (evenly-spaced buckets)
IQImportance-weighted quantization. Smarter: gives more precision to important weights. Usually better than Q at the same bit-width.
KK-quant: uses different block sizes for different layer types (attention gets different treatment than FFN)
_0Legacy format, fixed block size (usually 32)
_S / _M / _LSmall / Medium / Large. Controls how many weights get higher precision within the mix. Larger = more weights at higher bits = bigger file but better quality.
_XS / _XXSExtra-small variants (IQ only). Even more aggressive compression.

What happens to accuracy?

Not all layers degrade equally when quantized. The pattern:

Layer typeSensitivity to quantizationWhy
Feed-forward (FFN)🟢 LowMassive redundancy. You can cut precision aggressively before quality drops.
Attention (Q/K/V)🟡 MediumPrecision matters for comparing token similarities. Q4_K still works well though.
Embedding / Output🔴 HighThese define the token space. Errors here directly affect which words the model can distinguish.

This is why K-quants exist — they give more bits to sensitive layers and fewer to FFN layers. A Q4_K_M might use 6-bit for attention, 4-bit for FFN, and 6-bit for embeddings, averaging to ~4.6 bits overall.

📊 Real-world quality loss estimates
QuantPerplexity increaseReasoning qualityBest for
Q8_0~0.01-0.02Nearly identicalWhen you have the RAM and want safety
Q6_K~0.03-0.05Very slight degradationHigh-quality local inference on 16 GB GPUs

IQ3_XXS~0.30-0.60Significant degradation on reasoningMemory-constrained edge cases

Perplexity = how "surprised" the model is by text. Lower = better. These are approximate; actual values vary by base model and domain.

Your decision framework

Given your mission — running local models for agentic/thinking tasks — here's the heuristic:


If RAM ≥ model_size × 2.5:
    → Use Q8_0 or FP16. Don't think about it.

If RAM ≥ model_size × 1.8:
    → Use Q6_K. Tiny quality hit, big memory win.

If RAM ≥ model_size × 1.3:
    → Use Q4_K_M. The pragmatic default. Most people stop here.

If RAM is tight:
    → Try IQ4_XS first (smarter than Q4_K_M at same size)
    → Fall back to Q3_K_M only if you must
    → Avoid Q2/IQ2 for reasoning — the quality cliff is real

For agentic/general-purpose reasoning tasks:
    → Q8_0 or Q6_K is ideal (your Deep-SWE benchmarks confirm this)
    → Q4_K_M is acceptable for most things but expect subtle misses
    → Below Q4, reasoning degrades measurably
