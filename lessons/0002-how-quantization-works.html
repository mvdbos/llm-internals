Lesson 2: How Quantization Actually Works

Every weight in an LLM is stored as a number. At FP16, each number uses 16 bits (2 bytes). Quantization asks: what if we stored each weight in fewer bits?

Fewer bits means fewer distinct values — the fundamental tradeoff
FormatBits per weightDistinct valuesAnalogy
FP161665,536 (in any range)Measuring to the millimetre
Q8 (8-bit)8256Measuring to the centimetre
Q4 (4-bit)416Measuring to the nearest 10 cm
Q2 (2-bit)24Measuring to the nearest half-metre

Think of it like a photo. The original has millions of colours (FP16). Convert to 256 colours (Q8) — it looks almost the same. Convert to 16 colours (Q4) — slightly posterised but still recognisable. Convert to 4 colours (Q2) — barely recognisable. Quantization is the same tradeoff for model weights.

1. The core idea: block-wise scaling

You can't just round 0.0234 to 0 and −0.891 to −1 and call it done. That would destroy the weights. Instead, quantization groups weights into blocks and stores a shared scale factor and offset for each block:

original ≈ scale × quantized_value + offset

Where:

  • quantized_value = a small integer (0–15 for 4-bit, 0–255 for 8-bit)
  • scale = a single FP16 number that sets the spacing between levels (e.g. 0.1033)
  • offset = aligns the integer grid to the data (sometimes implemented via a zero point)

The scale and offset stay in full precision. Only the integers are cheap. Store 32 weights as 4-bit integers plus one FP16 scale and one FP16 offset:

Storage: 32 × 4 bits  +  16 bits  +  16 bits  =  160 bits  =  5.0 bits/weight

That's still 3.2× smaller than the original 16 bits/weight. With 128-weight blocks you approach 4.25 bits/weight — the overhead shrinks as blocks get larger. This is why effective bits per weight is usually higher than the nominal code width.

A concrete walkthrough

One weight through the encode–decode cycle:

Original weight: 0.31
Group range: min = −0.88, max = 0.67
Total span: 0.67 − (−0.88) = 1.55

Scale = 1.55 ÷ 15 = 0.1033 per level
Offset = −0.88

Encoding: 0.31 → integer
  int = round((0.31 − offset) ÷ scale) = round(1.19 ÷ 0.1033) = round(11.5)
  → stored as int 12

Decoding: int 12 → float
  value = scale × 12 + offset = 1.240 + (−0.88) = 0.36

Error: 0.36 − 0.31 = 0.05 (off by 5 hundredths)

A single weight is off by 0.05. That might seem like a lot — but the model has billions of weights, each carrying a tiny bit of redundant information. A small per-weight error doesn't catastrophically change the output; it adds a faint layer of noise. The model is robust to this because it learned to be during training (through techniques like dropout and weight decay).

2. Why blocks? Why not one scale per layer?

If scale factors are the trick, why not use them differently? There are only three options:

The three possible scaling strategies — only one works well
StrategyCompressionAccuracyVerdict
One scale per weight None — the scale is FP16, so you're back to 16 bits per weight Perfect Pointless
One scale per layer Maximum — one scale for millions of weights Terrible — weights in a layer span very different ranges; one scale can't capture both ends well Too coarse
One scale per block (e.g. 32 weights) Good — scale overhead is small relative to the weight savings Good — 32 neighbouring weights usually live in a similar range The sweet spot

Block size is a tunable parameter. A smaller group size adapts more closely to local value ranges, reducing reconstruction error — but it repeats metadata more often. A larger group amortises metadata over more values but can force outliers and small values to share one coarse mapping. The next sections make this precise.

3. One complete two-bit affine round trip

Consider one group containing eight real values:

original = [-1.00, -0.70, -0.40, -0.10, 0.20, 0.50, 0.70, 0.90]

Use four codes, 0, 1, 2, 3. The minimum is −1.00 and the maximum is 0.90, so the span is 1.90. Three equal intervals cover four levels:

scale = (maximum - minimum) ÷ (number_of_levels - 1)
      = (0.90 - (-1.00)) ÷ 3
      = 0.633333…

encode(x)      = round((x - minimum) ÷ scale)
reconstruct(q) = minimum + q × scale

4. Metadata changes the true storage cost

Eight two-bit codes need sixteen payload bits. But the group also needs enough metadata to reconstruct the values. For many groups:

total storage
= value_code_bits
+ scale/offset/codebook metadata
+ alignment and tensor metadata
+ any values kept in another format

A smaller group size can adapt more closely to local value ranges, often reducing reconstruction error, but it repeats metadata more often. A larger group amortizes metadata over more values but can force outliers and small values to share one coarse mapping.

That is why effective bits per weight can be higher than the nominal code width. The relevant calculation counts codes, metadata, and exceptions under one explicit accounting rule.

Storage accounting also records the data type of each component: packed-code storage, scale dtype, offset/bias or zero-point dtype, codebook dtype, and the dtypes of any unquantized exceptions. Two groups with the same nominal code width can therefore have different effective storage.

Symmetric parameterization

A symmetric linear mapping centers levels around zero and normally has no separate offset:

scale = max(|x|) ÷ q_max
q     = clip(round(x ÷ scale), −q_max, q_max)
x_hat = scale × q

Zero reconstructs exactly, while an asymmetric group may leave part of the code range poorly used.

Affine parameterization

An affine mapping shifts the grid with a zero point z (or an algebraically equivalent offset):

q     = clip(round(x ÷ scale) + z, q_min, q_max)
x_hat = scale × (q − z)

The scale controls spacing; the zero point controls alignment. They are distinct parameters. The toy calculation above writes the same idea using the group's minimum as an explicit reconstruction offset.

6. Measure error at more than one level

A low tensor reconstruction error is useful evidence, but it is not the final objective. For a dot product y = Σᵢ(wᵢaᵢ), replacing each weight with wᵢ + errorᵢ gives:

y_hat − y = Σᵢ(errorᵢ × activationᵢ)

Positive and negative terms can partly cancel, which helps explain why nonzero local errors need not produce the same output error. Cancellation is not guaranteed: a large activation can amplify one weight error, aligned errors can add, and outliers or sensitive directions can dominate an average. Later normalization, nonlinearities, residual paths, routing, and repeated generation can shrink, amplify, or redirect the perturbation.

7. Why approximation can be tolerable—and why that is not guaranteed

Neural computations combine many values, and trained representations can have redundancy. Small local perturbations may leave a selected output unchanged. But tolerance depends on the checkpoint, tensor role, magnitude distribution, architecture, mapping, workload, and runtime implementation. Outliers, repeated state use, routing boundaries, or decision thresholds can make some errors matter more than their average magnitude suggests.

The correct conclusion is not “models tolerate a particular bit width.” It is:

  1. verify the exact encoding and target;
  2. measure storage and runtime behavior;
  3. measure reconstruction or output diagnostics;
  4. evaluate paired tasks from the intended workload;
  5. report uncertainty and scope.
The same mechanics, different implementations

GGUF is an artifact/container family, llama.cpp Q8_0 and MLX affine 8-bit name different quantization schemes, MLX is a framework/ecosystem, and vLLM and SGLang are inference runtimes. These category labels do not define one interchangeable “8-bit” format. Lesson 3 covers the versioned implementation details.
