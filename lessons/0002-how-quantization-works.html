Lesson 2: How Quantization Actually Works

From 16-bit floats to 4-bit integers — and why the model still works. ~10 minutes.

What quantization does

Every weight in an LLM is stored as a number. At FP16, each number uses 16 bits (2 bytes). Quantization says: "what if we stored each weight in fewer bits?"

FormatBits per weightDistinct valuesAnalogy
FP161665,536 (in any range)Measuring to the millimeter
Q8 (8-bit)8256Measuring to the centimeter
Q4 (4-bit)416Measuring to the nearest 10 cm
Q2 (2-bit)24Measuring to the nearest half-meter
ELI5 version: You have a photo. The original has millions of colors (FP16). You convert it to 256 colors (Q8) — it looks almost the same. You convert to 16 colors (Q4) — it looks slightly posterized but still recognizable. You convert to 4 colors (Q2) — now it's barely recognizable. Quantization is the same tradeoff for model weights.

The core trick: scale factors

You can't just round 0.0234 to 0 and -0.891 to -1 and call it done. That would destroy the weights. Instead, quantization uses a simple formula:

original ≈ scale × quantized_value + zero_point

Where:

  • quantized_value = a small integer (0-15 for 4-bit, 0-255 for 8-bit)
  • scale = a single FP16 number telling you "how much each integer step is worth" (e.g. 0.1033)
  • zero_point = the integer that maps to zero (often the min value, like -0.88)

The scale and zero_point are stored in full FP16 precision. Only the integers are cheap. So if you store 32 weights as 4-bit integers plus one FP16 scale + one FP16 zero_point, you get:

Storage: 32 × 4 bits  +  16 bits  +  16 bits  =  160 bits  =  5.0 bits/weight

That's still 3.2× smaller than the original 16 bits/weight. The overhead from the scale factors shrinks as the block gets bigger — with 128-weight blocks you approach 4.25 bits/weight.

A concrete example

Let's walk one weight through the encode-decode cycle:

Original weight: 0.31
Group range: min = -0.88, max = 0.67
Total span: 0.67 - (-0.88) = 1.55

Scale = 1.55 / 15 = 0.1033 per level
Zero_point = -0.88

Encoding: 0.31 mapped to integer
  int = round((0.31 - (-0.88)) / 0.1033) = round(1.19 / 0.1033) = round(11.5)
  → stored as int 12

Decoding: int 12 back to float
  value = 0.1033 × 12 + (-0.88) = 1.240 + (-0.88) = 0.36

Error: 0.36 - 0.31 = 0.05  (off by 5 hundredths)

A single weight is off by 0.05. That might seem like a lot — but the model was trained with billions of weights, each carrying a tiny bit of redundant information. A small per-weight error doesn't catastrophically change the output; it adds a faint haze of noise that slightly degrades the signal. The model is robust to this noise because it learned to be during training (through techniques like dropout and weight decay).

🧠 Why this works

Weights within a small group tend to be in a similar range. A single scale factor can represent them all well without needing one per weight. It's like describing a set of heights in a classroom: you don't need every student's exact height in millimeters — "all between 1.50m and 1.85m, give me 16 buckets" is good enough. The model's knowledge is distributed across many weights, so no single weight being slightly off breaks anything.

Why blocks? Why not one scale per layer?

You might wonder: if scale factors are the trick, why not use them differently? There are only three options:

StrategyCompressionAccuracyVerdict
One scale per weight ❌ None — the scale is FP16, so you're back to 16 bits per weight ✅ Perfect Pointless
One scale per layer ✅ Maximum — one scale for millions of weights ❌ Terrible — the weights in a layer span very different ranges (some are near zero, others are large). One scale can't capture both ends well. Too coarse
One scale per block (e.g., 32 weights) ✅ Good — scale overhead is small relative to the weight savings ✅ Good — 32 neighboring weights usually live in a similar range The Goldilocks zone

Block size is a tunable parameter. Q8_0 uses blocks of 32. K-quants use a hierarchical "superblock" structure (256 weights split into sub-blocks) to get even finer-grained scaling where it matters. The larger the block, the better the compression but the coarser the mapping — 32 is a pragmatic balance.

Decoding the GGUF suffixes

When you download models from Hugging Face, you see filenames like Q4_K_M, Q8_0, IQ3_XXS. Here's what they mean:

SuffixBits/weightSize (7B model)What it does
F1616~14 GBFull half-precision. No quantization. The quality baseline.
Q8_08~7 GB8-bit with block size 32. Minimal quality loss (1-2%).
Q6_K6~5.3 GB6-bit with K-quant superblock. Very good quality.
Q5_K_M5~4.5 GB5-bit K-quant, medium size. Good balance point.
Q4_K_M4~3.8 GB4-bit K-quant, medium. The community sweet spot.
IQ4_XS4~3.5 GB4-bit importance-weighted. Smarter than Q4_K_M for some models.
Q3_K_M3~3.1 GB3-bit. Quality loss becomes noticeable for reasoning.
IQ3_XXS~3~2.9 GBAggressively compressed. Noticeable quality loss on complex tasks.
Q2_K2~2.5 GBVery aggressive. Usable only for simple tasks or tiny models.

What do the letters mean?

LetterMeaning
QTraditional linear quantization (evenly-spaced buckets)
IQImportance-weighted quantization. Smarter: gives more precision to important weights. Usually better than Q at the same bit-width.
KK-quant: uses different block sizes for different layer types (attention gets different treatment than FFN)
_0Legacy format, fixed block size (usually 32)
_S / _M / _LSmall / Medium / Large. Controls how many weights get higher precision within the mix. Larger = more weights at higher bits = bigger file but better quality.
_XS / _XXSExtra-small variants (IQ only). Even more aggressive compression.

What happens to accuracy?

Not all layers degrade equally when quantized. The pattern:

Layer typeSensitivity to quantizationWhy
Feed-forward (FFN)🟢 LowMassive redundancy. You can cut precision aggressively before quality drops.
Attention (Q/K/V)🟡 MediumPrecision matters for comparing token similarities. Q4_K still works well though.
Embedding / Output🔴 HighThese define the token space. Errors here directly affect which words the model can distinguish.

This is why K-quants exist — they give more bits to sensitive layers and fewer to FFN layers. A Q4_K_M might use 6-bit for attention, 4-bit for FFN, and 6-bit for embeddings, averaging to ~4.6 bits overall.

📖 A quick word on perplexity

Perplexity is the standard metric for measuring language model quality. It answers: "how surprised is the model by real text?" Lower is better. A perplexity of 10 means the model is as uncertain as if it were choosing randomly among 10 equally likely options at each word. A perfect model would have perplexity near 1. When we say quantization "adds 0.05 to perplexity," we mean the model became slightly more surprised — equivalent to having half an extra option in its guessing game.

📊 Real-world quality loss estimates
QuantPerplexity increaseReasoning qualityBest for
Q8_0~0.01-0.02Nearly identicalWhen you have the RAM and want safety
Q6_K~0.03-0.05Very slight degradationHigh-quality local inference on 16 GB GPUs

Your decision framework

Given your mission — running local models for agentic/thinking tasks — here's the heuristic:


If RAM ≥ model_size × 2.5:
    → Use Q8_0 or FP16. Don't think about it.

If RAM ≥ model_size × 1.8:
    → Use Q6_K. Tiny quality hit, big memory win.

If RAM ≥ model_size × 1.3:
    → Use Q4_K_M. The pragmatic default. Most people stop here.

If RAM is tight:
    → Try IQ4_XS first (smarter than Q4_K_M at same size)
    → Fall back to Q3_K_M only if you must
    → Avoid Q2/IQ2 for reasoning — the quality cliff is real

For agentic/general-purpose reasoning tasks:
    → Q8_0 or Q6_K is ideal (the case study in Lesson 3 shows why)
    → Q4_K_M is acceptable for most things but expect subtle misses
    → Below Q4, reasoning degrades measurably
