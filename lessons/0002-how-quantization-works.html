Lesson 2: How Quantization Actually Works

Every weight is stored as a number. At FP16, each number uses 16 bits. Quantization asks whether the model can store an approximation in fewer bits and still be useful for your work.

A simplified code-count comparison. Floating-point values are unevenly spaced, so FP16 is not 65,536 evenly spaced values in every range.
Stored representationCode bitsPossible codesPractical intuition
FP1616Many unevenly spaced floating-point valuesFine detail across a wide range
8-bit codes8256Smaller steps than lower-bit codes
4-bit codes416Fewer reconstruction choices
2-bit codes24Very coarse reconstruction choices

Think of a photo with fewer available colours: reducing choices saves space, but can erase detail. Quantization makes the same trade-off for stored weight values. The label tells you a storage scheme, not whether the resulting model is good enough for a particular workload.

1. Share a mapping across a block

Rather than rounding every value onto one global grid, many schemes group values and store a shared scale factor and offset. A pedagogical affine mapping is:

original ≈ scale × stored_code + offset

In this pedagogical example, scale and offset are stored in FP16. The low-bit code stores the position on the small grid; shared metadata says how to reconstruct it. Some implementations use an integer zero point instead of storing an offset directly.

One weight: encode → store → reconstruct → compare

Group range: minimum = -0.88, maximum = 0.67
Four-bit codes provide 16 levels, so there are 15 intervals.
scale  = (0.67 - (-0.88)) / 15 = 0.1033
offset = -0.88

Original weight = 0.31
stored_code = round((0.31 - offset) / scale)
            = round(11.5) = 12
reconstructed = scale × 12 + offset = 0.36
error = 0.36 - 0.31 = 0.05

The reconstructed value is close, not identical. That local error is useful for checking the mapping, but it does not prove that the model will preserve task quality. Tolerance depends on the model, which values were changed, the encoding, and the workload.

2. Why blocks are a trade-off

One mapping per value would repeat too much metadata. One mapping for a huge collection of values can be too coarse when its values have very different ranges. A group size sits between those extremes: smaller groups can fit local ranges more closely, while larger groups repeat metadata less often. Neither is automatically better; the scheme balances reconstruction detail against storage overhead.

One practical group-size trade-off
ChoiceLikely benefitLikely cost
Smaller groupMapping can follow a local range more closelyScale and offset metadata repeat more often
Larger groupMetadata is shared across more valuesOne coarse mapping may fit some values poorly

3. Count codes and metadata together

Suppose 32 values use four-bit codes plus one FP16 scale and one FP16 offset:

32 × 4 code bits + 16 scale bits + 16 offset bits
= 160 bits total
= 5.0 bits per stored value

The nominal code width is four bits, but reconstruction metadata raises the effective bits per weight. A real artifact can also contain alignment, tensor metadata, or values stored differently, so measure the actual candidate rather than treating a filename as a full memory forecast.

4. What this calculation can—and cannot—tell you

The round trip shows a local approximation. It can help explain why compact storage is possible. It cannot rank candidates by chat quality, coding reliability, speed, compatibility, or peak runtime memory. Those are separate questions that depend on the complete artifact, the inference engine, the hardware, and your intended work.

The same mechanics, different implementations

Different implementations give their artifacts, mappings, and runtimes different names. The shared idea here is compact codes plus reconstruction metadata. The next lesson explains how to identify a real candidate precisely before comparing it.

Takeaway and next step
