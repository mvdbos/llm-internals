Lesson 3: Choosing the Right Quantization

A practical guide to selecting quants for your hardware, your model, and your workload. ~10 minutes.

The quant selection problem

You've downloaded a model. You have limited VRAM or RAM. You see six different GGUF files — or your inference engine offers on-the-fly quantization. Which one do you pick?

The answer isn't "Q4_K_M is the sweet spot" (though it often is). The answer depends on three things:

  1. What are you using the model for? Chat? Code review? Agentic coding? Creative writing?
  2. What hardware do you have? How much VRAM/RAM is actually available after the OS and other apps?
  3. How big is the model? A 7B model at Q4 might be worse than a 13B model at Q3 — size and precision trade off against each other.

This lesson gives you a framework for making that decision systematically instead of guessing.

The three axes that matter

Axis 1: Task type — what are you asking the model to do?

Not all tasks care about weight precision equally. The more a task requires the model to maintain coherent state across many steps, the more precision matters.

Task typePrecision sensitivityWhy
Creative writing, brainstorming🟢 LowFuzziness can even add variety. Short context, no state to track.
General chat, Q&A🟢 LowSingle-turn or few-turn exchanges. Errors don't compound.
Summarization, classification🟡 MediumOne-shot tasks with clear input/output. Model doesn't need to remember past steps.
Code review, PR analysis🟡 MediumReasoning-heavy but bounded — one diff, one review. Medium context.
Multi-step reasoning, math🔴 HighEach step builds on the last. Small errors compound across the chain.
Agentic coding, long-horizon planning🔴 Very high100+ step loops. The model must track what it tried, what failed, and what's next. Precision loss in the KV cache snowballs.

Axis 2: Hardware budget — how much room do you actually have?

The rule of thumb from Lesson 2:

Model RAM ≈ parameters × bytes_per_weight  (+ 1-2 GB overhead)

FP16/bf16:  params × 2 bytes
Q8_0:       params × 1 byte
Q4_K_M:     params × 0.5 bytes
Q3_K_M:     params × 0.375 bytes

But don't fill your RAM completely. Leave headroom for:

  • The KV cache (grows with context length — can easily eat 2-4 GB on long conversations)
  • The OS and other running applications
  • Batch size if you're serving multiple requests

A safe target: model size ≤ 70% of your available VRAM/RAM. If you have 16 GB, aim for models under ~11 GB.

Axis 3: Model size — bigger models tolerate rougher quants

Larger models have more redundancy — the same knowledge is distributed across more weights. This means a 35B model at Q4_K_M might outperform a 27B model at Q8_0 on reasoning tasks, even though its weights are individually fuzzier. The extra 8 billion parameters compensate.

When comparing options, always test across model sizes, not just quants. The best setup might be: bigger model, more aggressive quant.

🧠 Quantization degrades reasoning quality, not capacity

A quantized model doesn't produce fewer thoughts — it produces slightly less coherent ones. With fuzzy weights, each reasoning step is a tiny bit off. Over a long chain, those tiny errors compound. The model doesn't run out of ideas; it just has subtly worse ideas, which lead it down paths that need more correction steps.

How to benchmark your own quants

The only way to know which quant works for your model on your hardware for your tasks is to test it. Here's a lightweight method:


For a model you want to run:

1. Pick ONE representative task from your actual workload
   - A coding issue, a long reasoning prompt, a multi-turn conversation
   - Not a synthetic benchmark — something you'd really use the model for

2. Run it through your baseline (FP16/bf16 if possible, or Q8_0 as reference)
   - Measure: steps taken, output quality, any looping or repetition

3. Run the SAME task through each candidate quant
   - Compare step count, output coherence, and any signs of looping

4. Interpret:
   Steps up <10% and output looks the same    → quant is safe for this task type
   Steps up 10-30% or minor quality drop      → borderline, test 2 more tasks
   Steps up >30% or visible looping           → quant is too aggressive

5. Test on 2-3 diverse tasks before committing to a quant for daily use

Don't benchmark one task and call it done. A quant that works perfectly for creative writing might fall apart on multi-step reasoning.

Case study: Agentic coding on Apple Silicon

To make this concrete, here's a real benchmark. A developer ran the same software engineering tasks (fixing real GitHub issues — database ORM bugs, HTTP parsing edge cases, API deprecation handling) through a 27B model at both bf16 and 8-bit precision:

Metricbf168-bitDelta
Avg steps per task74116+57%
Thought duplicationNone (ratio = 1.0)Systematic (ratio < 1.0)Qualitative cliff

The 8-bit model took 57% more steps to accomplish the same work — not because it was slower per step, but because it kept repeating itself. It would try an approach, forget what it did, and try again. Or loop on similar-looking reasoning without making progress.

🔍 The KV cache snowball

Here's why quantization hits long agentic runs especially hard:

  1. Step 1: The model processes token 1 using quantized attention weights. The computed attention scores are slightly off — maybe 0.73 instead of the true 0.78.
  2. KV cache stores it: The slightly-wrong attention Key and Value vectors are saved. There's no correction pass — whatever the model computed gets stored.
  3. Step 2: The model now attends to the stored representations from step 1, which are already degraded. The attention computation for step 2 works with fuzzy inputs, so its output is slightly more degraded.
  4. Steps 3–100: This compounds. By step 50, the KV cache contains 50 tokens of increasingly noisy representations. The model tries to recall "what I tried 30 steps ago," but that memory is blurry. It can't reliably distinguish "already tried that" from "considered that but haven't tried it." So it tries again. And again.

With bf16, the per-step error is small enough that the compound degradation across 100 steps is manageable. With 8-bit, each step introduces enough error that by step 50 the KV cache is meaningfully degraded.

This is also why short conversations (5-10 turns) show almost no quality difference between Q8 and bf16 — the compounding hasn't had time to accumulate.

Why throwing more tokens at it doesn't help

A natural instinct: "if the quantized model is struggling, give it more thinking budget — let it think longer." It doesn't work. Increasing the thinking budget on a quantized model just produces more fuzzy thoughts, not better ones. The model wasn't running out of space to think; it was producing lower-quality thoughts to begin with.

This is the core insight: quantization degrades the quality of each thought, not the quantity. You can't compensate for fuzzy weights with more tokens — you just get more fuzz.

📊 The bigger-model tradeoff

When RAM is fixed, you have a choice: run a smaller model at higher precision, or a larger model at lower precision. A 35B at Q4_K_M might outperform a 27B at Q8_0 on reasoning — the extra parameters compensate for fuzzier weights. Always benchmark both options. Don't assume higher precision always wins.
