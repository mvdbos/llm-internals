💡 Why this matters differently for you

Unlike most people downloading GGUF files from Hugging Face, you quantize at runtime via oMLX. This means you can switch between bf16, 8-bit, and 4-bit without re-downloading. It also means you're using oMLX's quantization implementation — which may differ subtly from llama.cpp's GGUF quants. The principles are the same, but your control is finer-grained.

What your benchmarks show: bf16 vs 8-bit

You ran the same Deep-SWE tasks (real GitHub issues like fixing drizzle-orm window functions, FastAPI deprecation handling, httpx multipart parsing) through both Qwen3.6-27B-bf16 and Qwen3.6-27B-8bit. Here's what happened:

The step increase isn't because 8-bit is "slower per step" — it's because the model repeats itself. It tries something, forgets what it did, and tries again. Or it gets stuck in a loop of similar-looking reasoning. Each duplicate thought burns steps without progress.

🔍 What's actually happening inside: the KV cache snowball

To understand why quantization causes loops in long agentic runs, you need to follow one chain of events:

  1. Step 1: The model processes token 1 using quantized attention weights. The result is slightly off — maybe the computed attention score between two tokens is 0.73 instead of the true 0.78.
  2. KV cache stores it: The slightly-wrong attention Key and Value vectors for token 1 are saved into the KV cache. There's no "correction pass" — whatever the model computed is what gets stored.
  3. Step 2: Now the model processes token 2. It attends not just to token 2's own input, but to the stored Keys and Values from token 1 — which are already slightly degraded. The attention computation for token 2 is now working with fuzzy inputs, so its output is slightly more degraded than token 1's.
  4. Steps 3–100: This compounds. By step 50, the KV cache contains 50 tokens' worth of increasingly noisy representations. The model tries to attend to "what I did 30 steps ago," but that memory is blurry. So it does something similar-but-not-quite-right. Then it corrects. Then it overcorrects. Then it looks like it's looping.

With bf16, the per-step error is small enough that the compound degradation across 100 steps is still manageable. With 8-bit, each step introduces enough error that by step 50 the KV cache is meaningfully degraded — the model can't reliably distinguish "I already tried that approach" from "I considered that approach but haven't tried it." So it tries again. And again.

This is also why short conversations (5-10 turns) show almost no quality difference between Q8 and bf16 — the compounding hasn't had time to accumulate. The degradation only becomes visible when you push into long-horizon reasoning.

Why thinking budget doesn't fix it

You tested this hypothesis: maybe the 8-bit model just needs more thinking tokens? The answer was no. Increasing the thinking budget on the 8-bit model didn't help — it just produced more duplicated thoughts. The model wasn't running out of space to think; it was producing lower-quality thoughts to begin with.

🧠 Key insight: Quantization degrades reasoning quality, not capacity

The model can still "think" for the same number of tokens. But with fuzzy weights, each thought is slightly less coherent. Over a long chain, those slight degradations compound. A model at Q4 doesn't run out of ideas — it just has subtly worse ideas, and those worse ideas lead it down paths that need more correction steps.

A systematic approach to choosing quants

Instead of guessing, here's a lightweight benchmarking procedure:


For a new model you want to run:

1. Run the SAME task through bf16 and your candidate quant
   - Pick a representative task from your workload (e.g., one Deep-SWE issue)
   - Compare: steps taken, unique_thoughts/total_thoughts ratio, final outcome

2. If steps increase by <10% and thoughts stay clean → the quant is safe
   If steps increase by 10-30% and thoughts show some duplication → borderline
   If steps increase by >30% and thoughts are messy → the quant is too aggressive

3. Test on 2-3 diverse tasks before committing to a quant for production use

4. For agentic workloads specifically: privilege precision.
   The cost of a bad step (wrong file edited, bad diagnosis) is much higher
   than the cost of loading a slightly larger model.

The bigger picture: when does precision actually matter?

Not all "intelligence" is equally sensitive to weight precision. Think of it as a spectrum:

CapabilityPrecision sensitivityWhy
Factual recall🟢 LowFacts are encoded redundantly across many weights. Small errors cancel out.
Language fluency🟢 LowGrammar and style are over-learned; robust to noise.
Pattern matching🟡 MediumNeeds enough precision to distinguish similar-but-different patterns.
Multi-step reasoning🔴 HighEach step depends on the previous one. Small errors compound.
Long-horizon planning🔴 Very highThe model must maintain a coherent "state" across many operations.

Your Deep-SWE benchmarks hit the bottom two rows — multi-step reasoning over long horizons. That's the worst case for quantization. A creative writing use case hits the top two — the best case.

📊 The 35B question

You also run Qwen3.6-35B. Larger models are more robust to quantization because they have more redundancy. A 35B at Q4_K_M might outperform a 27B at Q8_0 on reasoning tasks, simply because the extra parameters compensate for the fuzzier weights. When you benchmark, always compare across model sizes, not just within one model at different quants. The best setup might be: bigger model, more aggressive quant.
