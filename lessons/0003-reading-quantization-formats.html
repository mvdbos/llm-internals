Lesson 3: Reading Quantization Formats

Lesson 2 gave you implementation-independent mechanics. Real downloads add a naming problem: one short label can refer to an artifact type, a tensor encoding, a model-file recipe, or a precision category. This lesson gives each name one job.

1. Read the implementation stack from source state to device

A comparison begins with the same source checkpoint. Conversion produces a model artifact. An inference engine loads it, an execution backend schedules operations, and a device-specific kernel executes each supported tensor layout.

Checkpoint
Learned source state and tokenizer
Artifact
Packaged tensors and metadata
Encoding/recipe
Representation assigned to each target tensor
Engine/backend
Loader, scheduler, cache, and operations
Kernel/hardware
Actual instruction path and memory system

Skipping a layer creates false equivalences. Two files can start from the same checkpoint but use different tensor targets. Two encodings with similar nominal width can require different metadata. The same artifact can run differently across engine versions or hardware.

Suffix and importance-matrix boundary: llama.cpp labels ending in _S, _M, or _L distinguish versioned model-file recipe variants; the letters do not encode one universal tensor-by-tensor bit assignment. The pinned quantizer can also consume an importance matrix through --imatrix. Record how that matrix was generated, its data scope, quantizer revision, and every tensor override instead of treating “IQ” or a suffix as a quality guarantee.

3. A name is incomplete without identifying what changed

The quantized target might be all eligible weights, selected weight tensors, activations, or retained cache/state. Weight-only conversion is common, but it does not follow from “8-bit” alone. Read artifact metadata and the conversion code or command.

Minimum description

Prefer “weight-only affine 8-bit, group size 64, with named tensor exceptions” over “the 8-bit model.” The first description can be checked; the second cannot.

4. Nominal code width and total model storage

A group size or block size tells you how many values share metadata. The payload width does not include that metadata. A model-file mixed-precision recipe can also assign different encodings to different tensors.

Effective bits per weight counts the declared model tensor bytes, including encoding metadata and mixed/unquantized tensor storage, divided by parameter count. It is an accounting measurement, not a quality score.

5. Use the course's five-step decision procedure

  1. Identify the backend and quantized targets. Establish source provenance, artifact format, encoding/recipe, target tensors, engine/backend, kernels, and hardware.
  2. Calculate memory. Separate artifact bytes, loaded tensors, retained state, measured peak, and capacity headroom.
  3. Shortlist formats that fit. Keep only candidates with correct loader/kernel support and measured memory/speed compatible with the deployment constraints.
  4. Evaluate the exact workload. Compare paired task outcomes under controlled settings; neither a suffix nor file size supplies a quality ranking.
  5. Choose the Pareto point. Apply hard constraints first, then choose among the remaining nondominated measured trade-offs and state uncertainty.

Kernel support can reverse a size-only prediction

A compact tensor encoding reduces memory traffic only if the execution path can exploit it. A fallback may unpack to a wider temporary representation, move work to another device, or use a slower kernel. Therefore measure both memory and speed on the intended stack.
