Lesson 4: Measuring Quantization Quality

Format labels tell you what changed. They do not tell you whether the change matters for your workload. A useful evaluation compares matched conditions, measures accuracy and efficiency, and keeps explanations separate from observations.

1. Separate four kinds of statement

Scroll horizontally to compare statement types.

Mechanism, implementation detail, observation, and hypothesis require different evidence
Statement typeExampleRequired support
MechanismRestricted codes plus reconstruction metadata introduce approximation error.Mathematical derivation or high-trust technical source
Implementation detailA pinned converter leaves selected tensors at another precision.Pinned code, documentation, command, and artifact metadata
Empirical observationCondition B used less peak memory in this experiment.Recorded measurements under a declared protocol
HypothesisA particular tensor's error caused a task failure.Explicit label plus a new test that could distinguish the explanation

Do not turn temporal order into causation: “the quantized run looped, therefore quantization damaged attention” skips every competing explanation. The looping is an observation; the proposed cause is a hypothesis.

2. Define the decision before collecting data

Write one sentence: “Choose among candidate artifacts for this workload subject to this memory boundary, while optimizing these quality and speed outcomes.” That sentence determines the tasks, metrics, and measurement boundary.

Start from a pinned checkpoint. If two conditions differ in source model, tokenizer, agent prompt, tools, or environment, the comparison is not an isolated precision test.

3. Build matched conditions

Scroll horizontally to inspect all controls.

Hold these fields fixed unless the experiment explicitly studies them
FieldControlFailure if uncontrolled
Model identitySame source revision, tokenizer, architecture, and generation configurationCheckpoint capability is confounded with quantization
ConversionRecord each artifact's exact target, mode, grouping, recipe, and exceptions“Precision” conditions may alter different tensors
ExecutionSame inference engine, backend, hardware, and engine settingsKernel/runtime differences are mistaken for model differences
Agent/harnessSame version, system/task prompts, tools, step limit, timeout, and environment imageScaffolding differences change success and cost
GenerationSame context limit, output/token budget, sampling parameters, stop conditions, and seed policyOne condition receives more opportunity or different randomness
TaskSame task instance and verifier revision in each pairTask difficulty is confounded with the condition

If a setting cannot be matched, preregister it as a limitation. “Default settings” is not enough because defaults can differ across engine versions or model configurations.

4. Pair tasks and repeat stochastic trials

A paired design runs every task under every condition. Pairing controls for task difficulty because the comparison uses within-task differences.

for each task:
    for each preregistered repeat:
        run condition A
        run condition B
        retain both results, including failures

Randomize or alternate condition order when warm-up, thermal state, cache state, or time trends could matter. Run one warm-up policy consistently. Never run memory-heavy conditions in parallel when parallelism changes memory pressure or causes interference.

Repeat counts should be chosen before results are visible. Multiple repeats estimate stochastic variation; they do not turn a small, convenience-selected task set into a universal benchmark.

5. Preregister missing-data and exclusion rules

Failures are data. Before execution, define:

  • what counts as timeout, infrastructure failure, invalid patch/output, or verifier failure;
  • whether and when an infrastructure failure is rerun;
  • how a missing pair affects paired summaries;
  • which runs are excluded from latency or token summaries and why;
  • how partial task completion is scored;
  • which task/repeat IDs must exist before publication.
No post-hoc cleanup

Do not remove a slow, failed, or looping run because it “looks abnormal.” Apply the preregistered rule and publish counts of included, excluded, missing, and failed runs.

6. Measure accuracy, efficiency, and behavior separately

Scroll horizontally to compare metric roles.

Metric families answer different decision questions
FamilyPossible measuresInterpretation guardrail
Task accuracyExternal verifier pass/fail, rubric score, exact match, task-specific testsPrimary when it directly reflects the intended workload
Predictive diagnosticsLoss or perplexity on a named datasetUseful diagnostic; not a substitute for every downstream behavior
EfficiencyArtifact bytes, loaded/peak memory, prefill/decode latency, tokens, steps, wall timeState the process/device boundary and include failed runs appropriately
Behavior diagnosticsTool errors, retries, repetition under a preregistered detector, termination reasonSecondary unless validated; do not infer mechanism from a label

Report counts and denominators. For paired binary outcomes, show how often A-only passes, B-only passes, both pass, and neither passes. For continuous metrics, report per-pair deltas and distribution/uncertainty, not only two pooled averages.

7. Analyze without overstating

  1. Validate that every expected task × repeat × condition cell exists.
  2. Confirm parity fields and provenance before viewing outcomes.
  3. Compute paired task-quality results first.
  4. Compute efficiency and behavior diagnostics with preregistered inclusion rules.
  5. Report uncertainty and sensitivity to missing/excluded data.
  6. Label causal explanations as hypotheses unless a follow-up experiment isolates them.

One controlled matrix can support a decision for its checkpoint, artifacts, engine/backend, hardware, tasks, and settings. It cannot establish a universal quantization ranking.

8. Connect the experiment to the decision procedure

This lesson implements step 4: evaluate the exact workload. The output should be a table of supported candidates with measured quality, latency, and memory—not a slogan. Lesson 5 uses those results to choose a Pareto point and state uncertainty.

Quiz: protect the comparison

1. Why run every task under every condition?

2. A quantized run repeats a tool call. What can you claim immediately?

3. When should missing-data rules be chosen?
