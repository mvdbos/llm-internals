Quantization decision checklist

Use these five steps in order. A candidate is not “best” in isolation: it is suitable only for a checkpoint, engine/backend, hardware target, memory boundary, and workload.

1. Identify the backend and quantized targets

  • Pin the source checkpoint and tokenizer revision.
  • Name the artifact/container and conversion revision.
  • Name the inference engine, backend, version, and hardware.
  • List each quantized target separately: weights, activations, retained cache/state, embeddings/output, or selected tensors.
  • Record mode, code width, group/block size, calibration or importance procedure, per-tensor exceptions, and model-file mixture recipe.

2. Calculate memory

  • Record artifact bytes; do not call them runtime memory.
  • Estimate or measure loaded parameter allocation. If estimating, use effective bpw—not nominal code width.
  • Estimate architecture-appropriate retained state. Use a grouped-query-aware KV formula only for conventional cached attention; use a different state model for recurrent, state-space, linear-attention, or hybrid architectures.
  • Add temporary activations, workspaces, allocator/runtime overhead, and explicit headroom.
  • Verify with a peak measurement under the intended batch, prompt length, and generated length.

3. Shortlist formats that fit

  • Eliminate candidates that exceed the measured memory boundary with headroom.
  • Confirm the pinned engine can load every tensor type and metadata field.
  • Confirm native kernels exist for the target hardware; record unpacking, dequantization, copy, or device fallbacks.
  • Measure prefill and decode latency/throughput. A smaller artifact is not automatically faster.
  • Keep multiple candidates when their memory/speed differences overlap measurement noise.

4. Evaluate the exact workload

  • Pair the same task instance across conditions and hold the agent/harness, tools, prompts, token/context limits, sampling, environment, and scoring fixed.
  • Use task-specific quality metrics and an external verifier where possible; do not replace workload evidence with perplexity alone.
  • Repeat stochastic runs and retain all failures. Report paired deltas, variation, and uncertainty.
  • Preregister exclusions, timeouts, missing-data handling, and any secondary diagnostics.
  • Keep qualitative traces private when they contain prompts, patches, paths, or exact private identities; publish only sanitized aggregate evidence.

5. Choose the Pareto point

  • Plot or tabulate quality, latency, and peak memory for every surviving candidate.
  • Discard candidates dominated on all declared objectives.
  • Choose among the remaining Pareto points using the workload's actual constraints—not a universal format ranking.
  • State residual uncertainty, version scope, and the conditions that would trigger re-evaluation.

Release check

  • Every implementation claim links to a pinned canonical source.
  • Every numerical result names model/artifact scope and measurement method.
  • Hypotheses are labeled and kept separate from observations.
  • Public artifacts use stable aliases and expose no private identity, prompt, patch, path, or raw trajectory.
