Quantization decision checklist

Use this after Lesson 5. The lessons teach the supporting ideas in a beginner-friendly order. When you evaluate real candidates, use this checklist in the order shown.

A candidate is not “best” in isolation. It is suitable only for a checkpoint, inference engine/backend, hardware target, memory boundary, and workload.

1. Describe each candidate completely

Do not rely on a filename or one informal bit label. For every candidate, record:

checkpoint
+ artifact/container
+ quantized target
+ encoding/recipe and grouping
+ runtime/backend version

2. Check fit with useful context and headroom

  • Record artifact size, but do not call it runtime memory.
  • Measure loaded or idle memory under the named runtime and loading policy.
  • Run a representative prompt length, generated length, and batch.
  • Record peak memory and remaining headroom.
  • Remove candidates that do not fit the context and headroom you actually need.

Lesson 3 explains why file size, loaded weights, retained state, and temporary runtime allocations are different buckets. If architecture-specific calculations are useful, use the runtime-state reference; otherwise prefer a representative peak measurement.

3. Shortlist compatible candidates

  • Keep only candidates the intended runtime can load and run correctly on the intended hardware.
  • Keep only candidates that fit with the context and headroom from Step 2.
  • Measure representative generation speed rather than assuming a smaller artifact is faster.
  • Keep two or three candidates when quality is still unknown.

If a candidate loads but is unexpectedly slow or uses unexpectedly high memory, consult the optional format and runtime reference for implementation-level diagnostic detail.

Measure direct task quality first, then interpret memory and speed with that quality. For example, use verified tests for coding, accuracy or F1 for classification, factuality criteria for summarization, a defined rubric or blinded preference for writing, or externally verified completion for agentic work. 5. Choose the supported trade-off

  • Apply hard constraints first: compatibility, useful context, headroom, and minimum acceptable quality.
  • Compare the surviving candidates on task quality, peak memory, and speed.
  • Choose the trade-off that best suits the workload—not a universal format winner.
  • Record what is still uncertain and what would make you revisit the decision.

Optional depth

You do not need this material for every local-model choice. Use it when the decision is high-stakes, a candidate behaves unexpectedly, or you need to make an implementation-specific or public claim.

  • Runtime-state reference — architecture-specific retained-state accounting and formulas.
  • Format and runtime reference — version-pinned encoding, grouping, metadata, and runtime/kernel details.
  • Controlled evaluation protocol — paired/repeated comparisons, exclusions, and evidence boundaries for public or causal claims.

Red flags

  • A filename or bit label is treated as a complete candidate description.
  • Artifact size is treated as peak runtime memory.
  • A smaller artifact is assumed to be faster without measurement.
  • Different candidates are compared on different tasks or settings.
  • Failures are silently dropped from the decision.
  • A result for one model, runtime, and workload is described as a universal ranking.

Supporting references: artifacts, encodings, and runtimes · tensors and runtime state · controlled evaluation · evidence ledger.
