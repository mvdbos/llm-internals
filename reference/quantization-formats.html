Reference: quantization artifacts, encodings, and runtimes

A label such as “8-bit” is not enough to compare two artifacts. Identify the container, tensor encoding, quantized target, grouping, recipe, engine/backend version, and kernel support.

GGUF is not a runtime

GGUF means GGML Universal File. It is a model-artifact format with metadata and typed tensors. llama.cpp is one engine ecosystem that reads GGUF. Do not call an MLX artifact “GGUF,” and do not reuse a GGML tensor label for a different MLX encoding.

Nominal code width is not effective bits per weight

A low-bit code stores the per-value symbol. Real encodings also store block/group metadata and can leave selected tensors at other precisions.

raw_block_bpw = 8 × bytes_in_one_encoded_block ÷ values_in_block

model_effective_bpw = 8 × counted_model_tensor_bytes ÷ declared_parameter_count
  • Raw-block bpw includes the packed values and metadata in one block structure.
  • Model-level effective bpw additionally reflects the recipe's tensor mixture and unquantized exceptions under a declared accounting rule.
  • Artifact bytes can add headers, metadata, alignment, tokenizer files, and other packaging.
  • Runtime memory can differ again because an engine may map, unpack, align, or duplicate tensors.

Model-level mixture recipes

A recipe can inspect tensor name/category, architecture, layer position, expert/GQA properties, importance data, and fallback support before assigning an encoding. Therefore:

  • Do not translate _M into a universal rule such as “attention uses 6-bit and FFN uses 4-bit.”
  • Do not assume two files with the same recipe label have identical effective bpw across architectures.
  • Record the quantizer revision, command/parameters, calibration or importance data, source checkpoint, and resulting per-tensor metadata.
  • Treat model-card shorthand as a pointer to inspect, not complete provenance.

Record this comparison tuple

source checkpoint revision
→ artifact/container + revision
→ quantized targets
→ mode + code width + group/block size
→ calibration/importance recipe + exceptions
→ engine + backend + version
→ kernels + hardware
→ measured memory, speed, and paired task quality

Only compare candidates after every field is known or explicitly marked unknown. Choose among measured Pareto points rather than relying on a format ranking or universal default.
