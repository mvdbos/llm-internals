Reading quantization formats

Optional depth after Lesson 4. Use this page when you are looking at a real download and need to turn its label and metadata into a complete candidate description. You do not need to memorize a catalogue of format names.

Start by giving each name one job

A file name can mix several kinds of information. Separate them before comparing candidates:

Scroll horizontally to inspect every category.

Three representative names that answer different questions
ExampleWhat it identifiesWhat it does not establish
GGUFAn artifact/container: a packaged model file with tensors and metadata.The encoding of every tensor, runtime support, speed, or task quality.
Q4_K_MA model-file recipe: a rule for assigning encodings to model tensors.One universal four-bit tensor representation or a quality ranking.
MLX affine 8-bit/group 64A weight-quantization configuration: mode, code width, and group size.GGML Q8_0, runtime-state precision, or universal compatibility.

These labels can be related, but they are not synonyms. A recipe can use different encodings for different tensors. A compact artifact can still need a compatible inference engine, backend, and supported kernel path.

Read a candidate record

When a download or model card gives incomplete information, fill in this record. Mark an unknown field as unknown rather than guessing from a suffix.

checkpoint: [revision or unknown]
artifact/container: [for example, GGUF]
quantized target: [for example, selected weights]
encoding/recipe and grouping: [what the metadata actually says]
runtime/backend version: [the runtime you will use]
intended hardware: [the machine you will use]

The quantized target matters. “Eight-bit” could describe selected weights, another kind of tensor, or only part of the model. It does not automatically describe activations or retained runtime state.

Ask the compatibility questions

  • Can the intended runtime load this artifact and its tensor types?
  • Does the candidate fit with your required context and headroom?
  • Does it run at an acceptable speed on the intended hardware?
  • Does it preserve enough quality on the work you actually care about?

Loading successfully answers only the first question. The other three need the memory, speed, and workload checks from Lessons 3 and 5.

What a label cannot decide for you

  • Which candidate fits at your required context length.
  • Whether the runtime uses an efficient execution path on your hardware.
  • Whether a smaller artifact is faster in your workload.
  • Whether the output quality is acceptable for your task.

That is why the course does not rank format labels. Use the decision checklist to compare compatible candidates after you have described them accurately.

Where to go for implementation detail

If you need source-level information about a particular encoding, recipe, runtime version, or compatibility claim, use the evidence ledger. It keeps version-pinned source links and their scope next to the relevant claim. That detail is deliberately kept out of this page so that it remains useful while you are making a practical choice.
