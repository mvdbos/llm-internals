Reference: tensors, dataflow, and runtime state

This sheet separates what is stored in a model artifact from values created or retained while an inference engine runs it.

A complete next-token dataflow

Architecture scope: this trace describes a decoder-only, Transformer-style language model with conventional token embeddings, attention/FFN blocks, and a vocabulary projection. Other architectures may replace attention, change retained state, or combine recurrent and attention paths.

1. Text
Characters or bytes supplied to the tokenizer
2. Token IDs
Integers from the exact tokenizer revision
3. Embeddings
Rows looked up from an embedding tensor
4. Hidden states
Representations transformed by repeated model blocks
5. Logits
One raw score per vocabulary item
6. Selection
A decoding rule selects the next token ID
  1. Tokenize. Text becomes token IDs. Token boundaries and vocabulary size belong to the tokenizer/checkpoint pair.
  2. Embed. Each ID indexes an embedding vector, producing a tensor commonly shaped (batch, sequence, hidden_dimension).
  3. Transform. A conventional Transformer-style block normalizes representations, forms attention projections, combines retained and current key/value vectors under a causal mask, applies an output projection, and applies a position-wise FFN with residual paths. Exact order, normalization, attention type, gating, and dimensions come from the architecture.
  4. Project. The final hidden representation is mapped to vocabulary logits. The output tensor may be tied to the input embedding or stored separately.
  5. Select. Greedy selection, sampling, constraints, or another decoding rule chooses one token ID. The tokenizer decodes IDs back to text.
  6. Repeat. The new token is fed into the next generation step; architecture-specific retained state is reused or updated.

Causal attention, with its boundary stated

attention(Q, K, V)
= softmax((QKᵀ ÷ √dₖ) + causal_mask)V

For one conventional causal-attention operation, learned projections form query (Q), key (K), and value (V) tensors. The scaled query–key products are masked so a position cannot use future positions, normalized with softmax, and used to mix values. Grouped-query attention (GQA) and multi-query attention (MQA) can share fewer key/value heads across more query heads. This equation does not describe recurrent, state-space, or linear-attention state updates, and a real block also includes learned projections, residual paths, normalization, and often a gated FFN.

Estimate loaded weight storage

estimated_weight_bytes = parameter_count × effective_bits_per_weight ÷ 8

Use this only as an estimate. Effective bpw must include encoding metadata and any tensors stored at other precisions under a declared accounting rule. Artifact bytes can additionally contain headers and non-model files. Loaded memory can differ again because an engine may map, unpack, align, or copy tensors.

Worked weight estimate

Suppose a hypothetical 7-billion-parameter artifact has a measured model-level effective storage of 4.5 bits per parameter:

7,000,000,000 × 4.5 ÷ 8
= 3,937,500,000 bytes
= 3.9375 GB (decimal)
= 3.6671 GiB (binary)

This is an encoding estimate, not a loaded-memory guarantee. The same artifact may allocate more or less after mapping, alignment, unpacking, or copying.

Conventional attention KV-cache estimate

kv_bytes = batch
         × retained_sequence_length
         × layer_count
         × retained_kv_head_count
         × head_dimension
         × 2                 # keys and values
         × bytes_per_element

Architecture scope: this formula is for a decoder-only model that retains one key vector and one value vector per retained KV head, layer, and token. Read every dimension and cache dtype from the exact model and engine configuration.

  • Grouped-query attention (GQA): use the number of KV heads, not the larger number of query heads.
  • Multi-query attention (MQA): the retained KV-head count is often one, if that is what the configuration specifies.
  • Sliding-window or sink caches: use the number of positions actually retained per relevant layer, not automatically the full prompt length.
  • Quantized caches: include scales, offsets, block metadata, and any unquantized exceptions—not only the nominal code width.
  • Cross-attention, speculative decoding, batching, or beam search: account for each additional cache or sequence copy.

Worked KV example

For a hypothetical conventional decoder with batch 1, 8,192 retained tokens, 32 layers, 8 KV heads, head dimension 128, and two bytes per element:

1 × 8,192 × 32 × 8 × 128 × 2 × 2
= 1,073,741,824 bytes
= 1.00 GiB

This is the KV allocation under the formula only. It does not include weights, current-step activations, allocator overhead, or engine workspaces.

Combine the boundaries

If the hypothetical 7-billion-parameter artifact above also used the worked 1.00 GiB KV cache, the two estimated components would total:

3,937,500,000 + 1,073,741,824
= 5,011,241,824 bytes
= 5.0112 GB
= 4.6671 GiB

This subtotal still excludes current-step activations, workspaces, engine/allocator overhead, and safety headroom. A complete deployment decision therefore measures the real peak instead of presenting 4.6671 GiB as a capacity requirement.

Measurement checklist

  1. Record checkpoint revision, artifact revision, engine/backend version, and hardware.
  2. State the memory boundary: file bytes, loaded allocation, process resident memory, device allocation, or system-wide memory.
  3. Measure an idle loaded baseline and the peak under a declared batch, prompt length, and generated length.
  4. Record cache dtype, cache strategy, retained length, and architecture dimensions.
  5. Repeat measurements and report variation instead of treating one sample as exact.
